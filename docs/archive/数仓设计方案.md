# NewAPI 数据仓库设计方案

**设计时间**: 2026-03-13  
**设计目标**: 支持每日增量同步，满足现有分析需求，为后续扩展预留空间

---

## 📊 一、业务分析需求梳理

### 1.1 核心分析场景

基于现有代码和报告，识别出以下分析需求：

| 分析场景 | 数据需求 | 更新频率 |
|---------|---------|---------|
| **增长漏斗（AARRR）** | 用户注册、首次使用、留存、充值、复购 | 每日 |
| **DAU趋势** | 每日活跃用户数 | 每日 |
| **用户分层** | 按充值金额分层统计 | 每日 |
| **激活点分析** | 调用次数与充值转化率 | 每日 |
| **渠道归因** | 首次调用模型作为渠道代理 | 每日 |
| **Cohort留存** | 按周/月的用户留存率 | 每周 |
| **财务指标** | ARPU、ARPPU、总营收、转化率 | 每日 |
| **Whale分析** | Top用户消费分布 | 每日 |
| **复购分析** | 复购率、复购次数分布 | 每日 |
| **模型使用** | 模型偏好、使用时段、Token消耗 | 每日 |

### 1.2 数据源映射

| 数据源 | 用途 | 增量字段 |
|--------|------|---------|
| `/api/log/` (type=2) | API调用记录、消费记录 | `created_at` |
| `/api/log/` (type=1) | 充值记录 | `created_at` |
| `/api/user/` | 用户基础信息 | 需要确认 |
| `/api/channel/` | 渠道配置（维度表） | 全量同步 |
| `/api/group/` | 分组信息（维度表） | 全量同步 |
| `/api/token/` | Token信息（可选） | `accessed_time` |

## 🏗️ 二、数仓分层架构

采用经典的四层架构：**ODS → DWD → DWS → ADS**

```
┌─────────────────────────────────────────────────┐
│  ADS层 - 应用数据服务层                          │
│  (面向分析需求的汇总表)                           │
└─────────────────────────────────────────────────┘
                    ↑
┌─────────────────────────────────────────────────┐
│  DWS层 - 数据汇总层                              │
│  (按维度汇总的中间层)                            │
└─────────────────────────────────────────────────┘
                    ↑
┌─────────────────────────────────────────────────┐
│  DWD层 - 数据明细层                              │
│  (清洗和标准化后的明细数据)                       │
└─────────────────────────────────────────────────┘
                    ↑
┌─────────────────────────────────────────────────┐
│  ODS层 - 原始数据层                              │
│  (API获取的原始数据)                             │
└─────────────────────────────────────────────────┘
```

## 📦 三、详细表结构设计

### 3.1 ODS层（原始数据层）

**目标**: 存储API获取的原始数据，不做清洗，保留原始格式

#### 3.1.1 `ods_logs` - 日志原始表

```sql
CREATE TABLE ods_logs (
    id BIGINT PRIMARY KEY,
    user_id INT,
    created_at BIGINT,  -- Unix时间戳（秒）
    type INT,  -- 1=充值, 2=消费, 3=管理, 4=错误, 5=系统
    content TEXT,
    username VARCHAR(255),
    token_name VARCHAR(255),
    model_name VARCHAR(255),
    quota BIGINT,
    prompt_tokens INT,
    completion_tokens INT,
    use_time INT,
    is_stream BOOLEAN,
    channel INT,
    channel_name VARCHAR(500),
    token_id INT,
    group_name VARCHAR(255),
    ip VARCHAR(50),
    request_id VARCHAR(100),
    other TEXT,  -- JSON字符串
    sync_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP,  -- 同步时间
    sync_date DATE,  -- 同步日期（分区字段）
    INDEX idx_user_id (user_id),
    INDEX idx_created_at (created_at),
    INDEX idx_type (type),
    INDEX idx_sync_date (sync_date)
) PARTITION BY RANGE (sync_date);
```

**增量策略**: 
- 每日凌晨同步前一天的数据
- 使用 `created_at >= 上次最大created_at` 过滤
- 按 `sync_date` 分区，便于数据管理

#### 3.1.2 `ods_users` - 用户原始表

**增量策略**:

### 3.2 DWD层（数据明细层）

**目标**: 清洗和标准化数据，统一时间格式，解析JSON字段

#### 3.2.1 `dwd_logs` - 日志明细表

```sql
CREATE TABLE dwd_logs (
    log_id BIGINT PRIMARY KEY,
    user_id INT NOT NULL,
    log_date DATE NOT NULL,  -- 从created_at转换
    log_datetime DATETIME NOT NULL,  -- 从created_at转换
    log_type INT NOT NULL,  -- 1=充值, 2=消费, 3=管理, 4=错误, 5=系统
    username VARCHAR(255),
    token_id INT,
    token_name VARCHAR(255),
    model_name VARCHAR(255),
    channel_id INT,
    channel_name VARCHAR(500),
    group_name VARCHAR(255),
    
    -- 消费相关字段（type=2时有效）
    quota BIGINT,  -- 配额消耗
    prompt_tokens INT,
    completion_tokens INT,
    total_tokens INT,  -- prompt_tokens + completion_tokens
    use_time INT,  -- 使用时长（秒）
    is_stream BOOLEAN,
    
    -- 充值相关字段（type=1时有效）
    topup_amount DECIMAL(10,2),  -- 从content或other字段解析
    
    -- 其他字段
    ip VARCHAR(50),
    request_id VARCHAR(100),
    
    -- 从other字段解析的扩展字段
    billing_source VARCHAR(50),  -- wallet/credit等
    cache_tokens INT,
    cache_ratio DECIMAL(5,4),
    model_ratio DECIMAL(5,2),
    group_ratio DECIMAL(5,2),
    
    -- 时间维度
    log_hour INT,  -- 0-23
    log_weekday INT,  -- 0-6 (0=Monday)
    log_week INT,  -- 周数
    log_month INT,  -- 1-12
    
    sync_time TIMESTAMP,
    INDEX idx_user_id (user_id),
    INDEX idx_log_date (log_date),
    INDEX idx_log_type (log_type),
    INDEX idx_model_name (model_name),
    INDEX idx_channel_id (channel_id)
) PARTITION BY RANGE (log_date);
```

**数据清洗逻辑**:
1. 时间戳转换：`created_at` (秒) → `log_datetime` (DATETIME)
2. JSON解析：解析 `other` 字段，提取 `billing_source`、`cache_tokens` 等
3. 充值金额提取：从 `content` 或 `other` 字段解析充值金额（type=1时）
4. 维度字段：提取 `log_hour`、`log_weekday` 等时间维度

#### 3.2.2 `dwd_users` - 用户明细表

```sql
CREATE TABLE dwd_users (
    user_id INT PRIMARY KEY,

    quota BIGINT,  -- 配额余额
    used_quota BIGINT,  -- 已使用配额
    request_count INT,  -- 请求次数
    
    -- 推广相关
    aff_code VARCHAR(100),
```

## 🔄 四、增量同步策略

### 4.1 同步时间表

| 数据源 | 同步频率 | 同步时间 | 增量字段 |
|--------|---------|---------|---------|
| 日志数据 | 每日 | 02:00 | `created_at` |
| 用户数据 | 每日 | 02:30 | 全量同步 |
| 渠道数据 | 每日 | 03:00 | 全量同步 |
| 分组数据 | 每日 | 03:00 | 全量同步 |

### 4.2 日志数据增量逻辑

```python
# 伪代码
def sync_logs_incremental():
    # 1. 获取上次同步的最大created_at
    last_max_created_at = get_last_max_created_at()
    
    # 2. 计算同步时间范围（前一天）
    end_time = int((datetime.now() - timedelta(days=1)).timestamp())
    start_time = last_max_created_at
    
    # 3. 分页获取数据
    page = 1
    page_size = 500
    all_logs = []
    
    while True:
        logs = api_client.get_logs(
            page=page,
            page_size=page_size,
            start_time=start_time,
            end_time=end_time
        )
        
        if not logs:
            break
            
        all_logs.extend(logs)
        page += 1
        
        # 避免API限流
        time.sleep(0.5)
    
    # 4. 写入ODS层
    insert_into_ods_logs(all_logs)
    
    # 5. 更新同步记录
    update_sync_record('logs', max_created_at)
```

### 4.3 数据一致性保证

1. **幂等性**: 使用 `log_id` 作为主键，避免重复插入
2. **事务性**: 使用数据库事务保证数据一致性
3. **错误处理**: 记录同步失败的数据，支持重试
4. **监控告警**: 同步失败时发送告警

## 🛠️ 六、技术选型建议

### 6.1 数据库选择

**选项1: SQLite（推荐用于本地开发）**
- ✅ 轻量级，无需安装
- ✅ 适合中小规模数据（<100万条）
- ✅ 支持分区（通过应用层实现）
- ❌ 并发性能有限

**选项2: PostgreSQL（推荐用于生产）**
- ✅ 功能强大，支持分区
- ✅ 并发性能好
- ✅ JSON支持好
- ❌ 需要安装配置

**选项3: MySQL**
- ✅ 广泛使用
- ✅ 性能稳定
- ❌ JSON支持不如PostgreSQL

### 6.2 同步工具

- **Python + APScheduler**: 定时任务调度
- **Python + requests**: API调用
- **SQLAlchemy**: ORM框架，便于数据库操作

## 📋 八、增量表 vs 全量快照表分类

### 8.1 增量表（Append-Only）

**特点**: 只追加新数据，不更新历史数据

| 表名 | 层级 | 说明 | 增量字段 |

### 9.1 充值数据字段结构

**测试结果**:
- 充值金额在 `content` 字段中
- 格式示例: `"通过兑换码充值 ＄575.000000 额度，兑换码ID 603"`
- 需要正则表达式提取金额: `＄(\d+\.?\d*)` 或 `\$(\d+\.?\d*)`

**提取逻辑**:
```python
import re

def extract_topup_amount(content: str) -> float:
    """从content字段提取充值金额"""
    # 匹配 $数字 或 ＄数字
    pattern = r'[＄$](\d+\.?\d*)'
    match = re.search(pattern, content)
    if match:
        return float(match.group(1))
    return 0.0
```

### 9.2 用户注册时间

**测试结果**:
- `/api/user/` 接口**没有** `created_at` 字段
- 需要通过日志计算：用户首次出现在日志中的时间作为注册/首次使用时间

**计算逻辑**:
```sql
-- 从日志中计算用户首次使用时间
SELECT 
    user_id,
    MIN(log_datetime) as first_use_date
FROM dwd_logs
WHERE log_type = 2  -- 消费日志
GROUP BY user_id
```
