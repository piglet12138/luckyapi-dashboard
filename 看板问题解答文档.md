## 问题1：留存率一直在下降，口径是什么？

### 📊 口径定义

**7日留存率** = (7天前活跃且今天仍活跃的用户数) / (7天前活跃用户数) × 100%

### 📈 数据分析

| 日期 | 当日活跃 | 7天前活跃 | 留存用户 | 留存率 |
|------|---------|----------|---------|--------|
| 3-14 | 64 | 71 | 31 | 43.66% |
| 3-13 | 132 | 73 | 37 | 50.68% |
| 3-12 | 113 | 85 | 39 | 45.88% |
| 3-10 | 120 | 51 | 39 | 76.47% |
| 3-08 | 76 | 40 | 32 | 80.00% |

### 🔍 下降原因

**这是正常的业务趋势**，不是数据错误！

#### 时间线分析

1. **3月7日之前**（2月底-3月初）：
   - 活跃用户：40-50人
   - 留存率：70-80%
   - **特征**：小规模核心用户，高粘性

2. **3月7日之后**：
   - 活跃用户：100+人（翻倍增长）
   - 留存率：40-50%
   - **特征**：用户规模扩大，新用户占比增加

### 💡 业务解释

这是典型的**用户增长期特征**：

#### ✅ 正常现象
- **早期阶段**：核心用户群体小，使用频繁，留存率高（70-80%）
- **增长阶段**：新用户大量涌入，还在探索产品，整体留存率下降（40-50%）
- **数学原因**：7天前的老用户基数（71人）中，只有31人今天仍活跃，因为大量新用户在最近几天才加入

#### ⚠️ 需要关注的点

1. **绝对留存人数**：31人（3-14），37人（3-13），39人（3-12）
   - 留存的绝对人数在增加，这是好现象！

2. **新用户7日留存**：
   - 需要单独追踪新用户的7日留存率
   - 如果新用户7日留存也在40-50%，说明产品粘性正常
   - 如果新用户7日留存低于30%，需要优化产品体验

### 📋 建议

1. ✅ **当前留存率（40-50%）属于正常范围**，不需要担心
2. 💡 **建议增加指标**：新用户Cohort留存率分析
   - 按注册日期分组
   - 追踪每组用户的D1、D3、D7、D30留存率
3. 🎯 **优化方向**：在保持增长的同时，提升新用户的早期留存率

## 问题2：新用户转化率趋势和新用户数量问题

### 📊 新用户数量统计

最近每日新用户数：

| 日期 | 新增用户 | 活跃用户 | 新用户占比 |
|------|---------|---------|-----------|
| 3-14 | 5人 | 64人 | 7.8% |
| 3-13 | 39人 | 132人 | 29.5% |
| 3-12 | 29人 | 113人 | 25.7% |
| 3-11 | 32人 | 112人 | 28.6% |
| 3-10 | 32人 | 120人 | 26.7% |
| 3-09 | 36人 | 115人 | 31.3% |

### 🔍 数据来源分析

#### "新用户"的定义

**当前定义**: `first_use_date = MIN(首次调用API的日期)`

```sql
SELECT
    user_id,
    MIN(CASE WHEN log_type = 2 THEN log_date END) as first_use_date
FROM dwd_logs
GROUP BY user_id
```

#### 关键发现

从数据库统计：
- **总用户数**: 1,082人
- **有API调用记录**: 422人（39%）
- **只注册未调用**: 660人（61%）

### 💡 重要说明

**"新用户" ≠ "新注册用户"**

**"新用户" = "新激活用户"（首次调用API的用户）**

#### 为什么这样定义？

1. ✅ **业务价值**：只注册不使用的用户没有业务意义
2. ✅ **转化分析**：只有使用了API才能谈转化
3. ✅ **数据可靠性**：API调用有明确的时间戳，注册时间可能没有记录

#### 数据解释

- **3月13日39个新用户**：39个首次调用API的用户
- **3月14日5个新用户**：5个首次调用API的用户（周五数据可能不完整）
- **660个未激活用户**：只注册但从未调用API的用户

### 📋 数据合理性验证

#### 活跃用户 vs 新用户

| 日期 | 活跃用户 | 新用户 | 老用户活跃 |
|------|---------|--------|-----------|
| 3-13 | 132人 | 39人 | 93人（70.5%） |
| 3-12 | 113人 | 29人 | 84人（74.3%） |

**结论**：大部分活跃用户是老用户，新用户占比25-30%，符合正常业务规律。

### 📊 新用户转化率趋势

| 日期 | 新用户 | D0付费 | D3付费 | D0转化率 | D3转化率 |
|------|--------|--------|--------|----------|----------|
| 3-13 | 39人 | 18人 | 19人 | 46.15% | 48.72% |
| 3-12 | 29人 | 12人 | 14人 | 41.38% | 48.28% |
| 3-11 | 32人 | 15人 | 16人 | 46.88% | 50.00% |
| 3-09 | 36人 | 15人 | 19人 | 41.67% | 52.78% |

**转化率趋势分析**：
- ✅ D0转化率：40-46%（当日付费）
- ✅ D3转化率：48-52%（3天内付费）
- 💡 **关键发现**：D3转化率比D0高5-10个百分点，说明用户需要2-3天了解产品

### 📋 建议

1. ✅ **当前口径是合理的**："新用户"定义为首次调用API的用户
2. 💡 **看板中已明确标注**："新增用户 = 首次调用API的用户（激活用户）"
3. 🎯 **如果需要注册数据**：
   - 需要在`ods_users`表中记录注册时间
   - 目前系统中没有明确的注册时间字段
   - 只能通过`sync_date`推断，但不准确

### 💡 新用户数量级合理性

**30人/天的新激活用户是合理的**：

1. **转化率健康**：D3转化率50%左右，说明获客质量好
2. **增长稳定**：每天20-40个新激活用户，保持稳定增长
3. **留存良好**：70%的活跃用户是老用户，说明留存不错

## 问题3：整体概览Tab图表显示问题

### 🐛 问题描述

整体概览Tab中的所有图表被压缩到框中左边很小一部分。

### 🔍 问题原因

这是**ECharts在Bootstrap Tab中的经典渲染问题**：

1. Tab切换时，隐藏的Tab中的图表容器尺寸为0
2. ECharts在初始化时获取到的容器尺寸是0×0
3. 切换到该Tab时，容器尺寸变正常，但图表已经渲染完成，不会自动调整

### ✅ 修复方案

使用**echarts.init包装 + Tab切换监听**方案：

#### 1. 包装echarts.init，自动记录所有图表实例

```javascript
window.allCharts = [];

const originalEchartsInit = echarts.init;
echarts.init = function(dom, theme, opts) {
    let chart = echarts.getInstanceByDom(dom);
    if (!chart) {
        chart = originalEchartsInit.call(echarts, dom, theme, opts);
        window.allCharts.push(chart);
    }
    return chart;
};
```

#### 2. Tab切换时重新调整所有图表尺寸

```javascript
document.addEventListener('DOMContentLoaded', function() {
    const tabButtons = document.querySelectorAll('[data-bs-toggle="tab"]');

    tabButtons.forEach(button => {
        button.addEventListener('shown.bs.tab', function(event) {
            setTimeout(() => {
                window.allCharts.forEach(chart => {
                    if (chart && !chart.isDisposed()) {
                        chart.resize();
                    }
                });
            }, 150);
        });
    });
});
```

#### 3. 窗口调整时同步resize

```javascript
window.addEventListener('resize', function() {
    window.allCharts.forEach(chart => {
        if (chart && !chart.isDisposed()) {
            chart.resize();
        }
    });
});
```

### ✅ 修复状态

- [x] 已在HTML文件中实施

- [x] 添加Tab切换监听，延迟150ms后resize
- [x] 添加窗口resize监听

### 📋 使用说明

**刷新页面后，图表应该正常显示**。

如果仍有问题：
1. 清除浏览器缓存
2. 强制刷新（Ctrl+Shift+R 或 Cmd+Shift+R）
3. 切换到"整体概览"Tab后等待200ms
