## 1. 快速开始

### 1.1 访问数据看板

**本地访问**:

1. 打开命令行，进入项目目录
2. 启动本地服务器：
   ```bash
   cd dashboard
   python -m http.server 8000
   ```
3. 在浏览器中访问：`http://localhost:8000`

**生产环境**:

直接访问部署的看板URL（由IT团队提供）

### 1.2 看板概览

看板包含以下模块：

- **核心指标卡片**: DAU、总调用次数、累计收入、转化率
- **用户增长分析**: 用户增长趋势、DAU趋势
- **留存与转化**: 留存率趋势、收入趋势
- **用户分层**: 用户分层分布、激活点分析
- **渠道分析**: 渠道新增用户排行、渠道转化率对比
- **模型使用**: 模型调用排行

### 1.3 数据更新频率

- **更新时间**: 每日凌晨2点自动更新
- **数据延迟**: T+1（今天看到的是昨天的数据）
- **历史数据**: 保留全部历史数据

## 2. 数据看板使用

### 2.1 核心指标卡片

#### DAU（日活跃用户数）

**定义**: 当日有API调用行为的独立用户数

**如何解读**:
- 反映产品的用户粘性和活跃度
- 持续上升表示用户留存良好
- 突然下降需关注是否有系统故障或用户流失

**正常范围**: 60-130人/天

#### 总调用次数

**定义**: 当日所有用户的API调用总次数

**如何解读**:
- 反映产品使用强度
- 与DAU结合看人均调用次数
- 调用量大幅波动需关注原因

**正常范围**: 4,000-17,000次/天

#### 累计收入

**定义**: 从开始到当日的总充值金额

**如何解读**:
- 反映商业化效果
- 关注增长速度和趋势
- 结合用户数看人均贡献

**当前值**: $66,817

#### 转化率

**定义**: 当日活跃用户中付费用户的占比

**如何解读**:
- 反映付费意愿
- 高于30%表示转化良好
- 低于20%需优化付费引导

**正常范围**: 25-35%

### 2.2 用户增长分析

#### 用户增长趋势图

**展示内容**:
- 新增用户：每日首次使用的用户数
- 活跃用户：每日有调用行为的用户数
- 付费用户：每日有充值行为的用户数

**如何解读**:
- 新增用户稳定表示获客渠道健康
- 活跃用户/新增用户比值反映留存情况
- 付费用户增长速度反映商业化效果

**关注点**:
- 新增用户是否稳定增长
- 活跃用户是否保持在合理水平
- 付费用户占比是否提升

#### DAU趋势图

**展示内容**: 最近30天的日活跃用户数变化

**如何解读**:
- 上升趋势：产品吸引力增强，用户留存好
- 平稳趋势：用户规模稳定
- 下降趋势：需关注用户流失原因

**异常情况**:
- 周末DAU下降：正常（工作日使用为主）
- 突然大幅下降：可能是系统故障或重大事件

### 2.3 留存与转化

#### 留存率趋势图

**展示内容**:
- 7日留存率：7天前新增用户中今天仍活跃的比例
- 付费转化率：活跃用户中付费用户的比例
- 复购率：付费用户中多次充值的比例

**如何解读**:
- **7日留存率**:
  - >40%：优秀，用户粘性强
  - 20-40%：良好，需持续优化
  - <20%：较差，需改进产品体验
- **付费转化率**:
  - >30%：优秀
  - 20-30%：良好
  - <20%：需优化付费引导
- **复购率**:
  - >50%：优秀，用户满意度高
  - 30-50%：良好
  - <30%：需提升服务质量

#### 收入趋势图

**展示内容**: 累计收入随时间的变化

**如何解读**:
- 斜率反映收入增长速度
- 平缓表示增长放缓，需加强运营
- 陡峭表示增长加速，可能有大客户或活动

**关注点**:
- 收入增长率是否符合预期
- 是否有明显的增长拐点
- 与用户增长是否匹配

### 2.4 用户分层

#### 用户分层分布饼图

**分层标准**:
- **未付费**: 从未充值的用户
- **轻度用户**: 调用量 <100次
- **中度用户**: 调用量 100-1000次
- **重度用户**: 调用量 >1000次

**如何解读**:
- 重度用户占比反映核心用户规模
- 未付费用户占比反映转化空间
- 各层级用户数量变化反映用户成长路径

**运营策略**:
- **未付费用户**: 引导首次充值，提供优惠
- **轻度用户**: 提升使用频次，培养习惯
- **中度用户**: 推荐高级功能，促进升级
- **重度用户**: VIP服务，防止流失

#### 激活点分析图

**展示内容**: 不同调用次数区间的付费转化率

**关键发现**:
- **15次调用是激活临界点**
  - <15次：转化率<18%
  - 15-50次：转化率41.67%
  - >50次：转化率>63%

**如何应用**:
- 引导新用户快速达到15次调用
- 设计新手任务，奖励前15次调用
- 在15次调用后推送付费引导

### 2.5 渠道分析

#### 渠道新增用户排行

**定义**: 各渠道（首次使用模型）带来的新增用户数

**如何解读**:
- 识别高价值获客渠道
- 评估渠道投放效果
- 优化资源分配策略

**Top渠道**:
1. claude-sonnet-4-6: 171用户
2. claude-sonnet-4-5: 54用户
3. claude-haiku-4-5: 52用户

**运营建议**:
- 加大头部渠道投入
- 优化尾部渠道转化
- 测试新渠道可行性

#### 渠道转化率对比

**定义**: 各渠道用户的付费转化率

**如何解读**:
- 高转化率渠道：用户质量好，应加大投入
- 低转化率渠道：需优化引导策略或减少投入
- 结合用户量和转化率综合评估ROI

**高转化渠道**:
- gcp/gemini-3-pro: 100%
- gcp/gemini-3-flash: 81.67%
- claude-sonnet-4-5: 66.26%

**优化方向**:
- 分析高转化渠道的用户特征
- 复制成功经验到其他渠道
- 对低转化渠道进行A/B测试

### 2.6 模型使用

#### 模型调用排行

**展示内容**: 各模型的总调用次数Top 10

**如何解读**:
- 反映用户需求偏好
- 指导产品优化方向
- 支持定价策略调整

**运营应用**:
- 热门模型：保证稳定性，可适当提价
- 冷门模型：分析原因，考虑下线或优化
- 新模型：观察使用趋势，评估推广效果

## 3. 常用查询

### 3.1 使用SQLite查询

**连接数据库**:

```bash
sqlite3 newapi_warehouse.db
```

### 3.2 核心指标查询

#### 查看最新核心指标

```sql
SELECT
    stat_date as 日期,
    active_users as DAU,
    total_calls as 调用次数,
    total_revenue as 累计收入,
    arpu as 人均收入,
    conversion_rate as 转化率
FROM ads_daily_summary
ORDER BY stat_date DESC
LIMIT 1;
```

#### 查看最近7天趋势

```sql
SELECT
    stat_date as 日期,
    active_users as DAU,
    new_users as 新增用户,
    paying_users as 付费用户,
    total_calls as 调用次数,
    conversion_rate as 转化率
FROM ads_daily_summary
ORDER BY stat_date DESC
LIMIT 7;
```

### 3.3 用户分析查询

#### 查看用户分层统计

```sql
SELECT
    user_segment as 用户分层,
    COUNT(*) as 用户数,
    SUM(total_call_count) as 总调用,
    ROUND(AVG(total_call_count), 2) as 人均调用,
    SUM(total_topup_amount) as 总收入,
    ROUND(AVG(total_topup_amount), 2) as 人均收入
FROM dws_user_lifecycle
GROUP BY user_segment
ORDER BY user_segment;
```

#### 查看高价值用户

```sql
SELECT
    user_id as 用户ID,
    total_call_count as 总调用,
    total_topup_amount as 总充值,
    user_segment as 用户分层,
    first_use_date as 首次使用,
    last_use_date as 最后使用
FROM dws_user_lifecycle
WHERE total_topup_amount > 100
ORDER BY total_topup_amount DESC
LIMIT 20;
```

### 3.4 渠道分析查询

#### 查看渠道效果

```sql
SELECT
    first_model as 渠道,
    SUM(new_user_count) as 新增用户,
    SUM(paying_user_count) as 付费用户,
    ROUND(AVG(conversion_rate), 2) as 平均转化率,
    SUM(total_revenue) as 总收入,
    ROUND(SUM(total_revenue) / SUM(new_user_count), 2) as 用户价值
FROM ads_channel_attribution
GROUP BY first_model
ORDER BY 新增用户 DESC
LIMIT 10;
```

### 3.5 模型分析查询

#### 查看模型使用排行

```sql
SELECT
    model_name as 模型名称,
    SUM(call_count) as 总调用,
    SUM(user_count) as 使用用户数,
    ROUND(SUM(call_count) * 1.0 / SUM(user_count), 2) as 人均调用
FROM dws_model_daily
GROUP BY model_name
ORDER BY 总调用 DESC
LIMIT 10;
```

## 4. 业务指标说明

### 4.1 用户指标

| 指标 | 英文 | 计算公式 | 说明 |
|------|------|---------|------|
| 日活跃用户 | DAU | 当日有调用的独立用户数 | 反映产品活跃度 |
| 新增用户 | New Users | 当日首次使用的用户数 | 反映获客能力 |
| 付费用户 | Paying Users | 当日有充值的用户数 | 反映商业化效果 |
| 7日留存率 | 7-Day Retention | 7天前新增用户中今天仍活跃的比例 | 反映用户粘性 |

### 4.2 收入指标

| 指标 | 英文 | 计算公式 | 说明 |
|------|------|---------|------|
| 累计收入 | Total Revenue | 所有充值金额之和 | 反映总体商业化效果 |
| ARPU | Average Revenue Per User | 收入 / 活跃用户数 | 人均收入 |
| ARPPU | Average Revenue Per Paying User | 收入 / 付费用户数 | 付费用户人均收入 |
| LTV | Lifetime Value | 用户生命周期总价值 | 用户长期价值 |

### 4.3 转化指标

| 指标 | 英文 | 计算公式 | 说明 |
|------|------|---------|------|
| 付费转化率 | Conversion Rate | 付费用户 / 活跃用户 | 反映付费意愿 |
| 激活率 | Activation Rate | 激活用户 / 注册用户 | 反映产品吸引力 |
| 复购率 | Repurchase Rate | 多次充值用户 / 付费用户 | 反映用户满意度 |

### 4.4 使用指标

| 指标 | 英文 | 计算公式 | 说明 |
|------|------|---------|------|
| 总调用次数 | Total Calls | 所有API调用次数 | 反映使用强度 |
| 人均调用 | Calls Per User | 调用次数 / 用户数 | 反映用户活跃度 |
| Token使用量 | Token Usage | Prompt + Completion Tokens | 反映资源消耗 |

## 5. 数据更新

### 5.1 手动更新数据

如需手动更新看板数据：

```bash
# 1. 进入项目目录
cd /path/to/newapi_export

# 2. 运行导出脚本
python export_dashboard_data.py

# 3. 刷新浏览器查看最新数据
```

### 5.2 自动更新配置

**Linux/Mac (crontab)**:

```bash
# 编辑定时任务
crontab -e

# 添加以下行（每日凌晨2点执行）
0 2 * * * cd /path/to/newapi_export && python export_dashboard_data.py
```

**Windows (任务计划程序)**:

1. 打开"任务计划程序"
2. 创建基本任务
3. 触发器：每天凌晨2点
4. 操作：启动程序 `python.exe`
5. 参数：`export_dashboard_data.py`
6. 起始于：项目目录路径

### 5.3 数据更新验证

检查数据是否更新成功：

1. 查看看板右下角的"数据更新时间"
2. 检查核心指标是否为最新日期
3. 查看日志文件确认执行状态

## 6. 常见问题

### 6.1 看板相关

**Q: 看板打不开怎么办？**

A: 检查以下几点：
1. 确认本地服务器已启动
2. 确认端口8000未被占用
3. 尝试更换端口：`python -m http.server 8001`
4. 检查浏览器控制台是否有错误

**Q: 看板数据不更新怎么办？**

A:
1. 检查 `dashboard_data.json` 文件的修改时间
2. 手动运行 `python export_dashboard_data.py`
3. 清除浏览器缓存后刷新
4. 检查定时任务是否正常执行

**Q: 图表显示异常怎么办？**

A:
1. 检查浏览器控制台是否有JavaScript错误
2. 确认 `dashboard_data.json` 格式正确
3. 尝试使用其他浏览器
4. 检查网络连接（CDN资源加载）

### 6.2 数据相关

**Q: 为什么今天看不到今天的数据？**

A: 数据是T+1更新，今天看到的是昨天的数据。这是为了保证数据完整性和准确性。

**Q: 数据有延迟怎么办？**

A:
1. 检查源数据库是否正常
2. 检查ETL脚本是否执行成功
3. 查看日志文件排查错误
4. 联系技术团队协助

**Q: 发现数据异常怎么办？**

A:
1. 记录异常数据的具体情况（日期、指标、数值）
2. 使用SQL查询验证原始数据
3. 联系数据团队进行核查
4. 必要时重新运行ETL流程

### 6.3 查询相关

**Q: 如何导出数据到Excel？**

A:
```bash
# 使用SQLite命令导出CSV
sqlite3 newapi_warehouse.db
.mode csv
.output data.csv
SELECT * FROM ads_daily_summary;
.quit

# 然后用Excel打开 data.csv
```

**Q: 如何查询特定日期范围的数据？**

A:
```sql
SELECT *
FROM ads_daily_summary
WHERE stat_date BETWEEN '2026-03-01' AND '2026-03-14'
ORDER BY stat_date;
```

**Q: 如何计算自定义指标？**

A: 可以基于现有表进行SQL查询计算，例如：
```sql
-- 计算周平均DAU
SELECT
    AVG(active_users) as 周平均DAU
FROM ads_daily_summary
WHERE stat_date >= date('now', '-7 days');
```
