## 一、修复内容

### 1. 留存率计算逻辑修复 ✅

**问题描述**:
- 原逻辑：只统计7天前**新用户**的留存
- 导致结果：留存率异常低（1.41%、5.48%）

**修复方案**:
```sql
-- 修复前（错误）
retention_rate = 7天前新增且今天仍活跃的用户 / 7天前活跃用户

-- 修复后（正确）
retention_rate = 7天前活跃且今天仍活跃的用户 / 7天前活跃用户
```

**修复效果**:
| 日期 | 修复前 | 修复后 | 说明 |
|------|--------|--------|------|
| 2026-03-14 | 1.41% | 43.66% | 正常范围 |
| 2026-03-13 | 5.48% | 50.68% | 正常范围 |
| 2026-03-12 | 5.88% | 45.88% | 正常范围 |

**验证**:
```bash
# 2026-03-14的留存率验证
7天前(2026-03-07)活跃用户: 71人
7天前活跃且今天仍活跃: 31人
留存率: 31/71 = 43.66% ✅
```

### 2. 付费转化率计算逻辑修复 ✅

**问题描述**:
- 原逻辑：使用**累计付费用户** / 当日活跃用户
- 导致结果：转化率>100%（435.94%、204.55%）

**修复方案**:
```sql
-- 修复前（错误）
conversion_rate = 累计付费用户 / 当日活跃用户 * 100

-- 修复后（正确）
conversion_rate = 当日付费用户 / 当日活跃用户 * 100
```

**修复效果**:
| 日期 | 修复前 | 修复后 | 说明 |
|------|--------|--------|------|
| 2026-03-14 | 435.94% | 29.69% | 正常范围 |
| 2026-03-13 | 204.55% | 28.03% | 正常范围 |
| 2026-03-12 | 220.35% | 31.86% | 正常范围 |

**验证**:
```bash
# 2026-03-14的转化率验证
当日付费用户: 19人
当日活跃用户: 64人
转化率: 19/64 = 29.69% ✅
```

### 3. 渠道转化率计算优化 ✅

**问题描述**:
- 原逻辑：使用AVG(conversion_rate)对每天的转化率求平均
- 导致问题：小样本日期权重过大，不能真实反映整体转化率

**修复方案**:
```sql
-- 修复前（不准确）
conversion_rate = AVG(每日转化率)

-- 修复后（准确）
conversion_rate = SUM(付费用户) / SUM(新增用户) * 100
```

**修复效果**:
| 渠道 | 修复前 | 修复后 | 差异 |
|------|--------|--------|------|
| claude-sonnet-4-5 | 66.26% | 55.56% | -10.7% |
| gcp/gemini-3-flash | 81.67% | 80.0% | -1.67% |
| claude-sonnet-4-6 | 54.29% | 54.97% | +0.68% |

SELECT
    '7天前活跃用户' as metric,
    COUNT(DISTINCT user_id) as value
FROM dws_user_daily
WHERE stat_date = '2026-03-07'
-- 结果: 71人

UNION ALL

SELECT
    '7天前活跃且今天仍活跃',
    COUNT(DISTINCT d1.user_id)
FROM dws_user_daily d1
WHERE d1.stat_date = '2026-03-07'
  AND EXISTS (
      SELECT 1 FROM dws_user_daily d2
      WHERE d2.user_id = d1.user_id
        AND d2.stat_date = '2026-03-14'
  )
-- 结果: 31人
