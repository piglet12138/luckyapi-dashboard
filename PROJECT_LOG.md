# 项目进度记录

## 2026-03-28 数据同步问题修复

### 问题描述
看板显示26-27号数据异常，DAU和收入数据不正确。

### 问题排查

**根本原因**：
1. 同步逻辑bug：原始逻辑"连续5页已存在就停止"在手动补数据后会跳过中间数据
2. ID特性：API的ID按时间严格递增（每天的ID范围不重叠）
3. 修复尝试：改为"ID <= db_max_id就停止"，但在删除数据后会失败

**数据恢复过程**：
- 使用`sync_by_time.py`按时间范围补齐26号数据
- 运行`daily_update.py`重新同步并计算
- 最终数据完整：26号(17936条)、27号(14590条)、28号(4090条)

### 解决方案

**1. 创建repair_data.py（解耦修复脚本）**
- 检查ODS→DWD缺失并补齐
- 检查DWD→ADS缺失并重新计算
- 完全幂等，可重复运行

**2. 优化同步逻辑**
- 保留基于ID的停止判断（因为ID按时间递增）
- 修复边界条件处理

**3. 新增工具**
- `sync_by_time.py`：按时间范围同步数据
- `repair_data.py`：幂等的数据修复工具

### 当前状态
✅ 数据完整性已恢复
✅ 看板显示正常
✅ 修复工具已就绪

## 数据管道主流程

### 每日自动执行（cron: 0 2 * * *）

```
daily_update.py
├── 1. 增量同步API数据 (IncrementalSync)
│   ├── 查询数据库最大ID
│   ├── 从API获取新数据（按页）
│   ├── 插入到ODS层（ods_logs）
│   └── 停止条件：min_id <= db_max_id
│
├── 2. ODS → DWD (ODSToDWD)
│   ├── 增量转换：只处理新ID
│   ├── 数据清洗和脱敏
│   └── 插入dwd_logs
│
├── 3. DWD → DWS (DWDToDWS)
│   └── 汇总用户生命周期数据
│
├── 4. DWS → ADS (DWSToADS)
│   ├── 计算每日漏斗指标
│   └── 计算留存、分层等
│
└── 5. 导出看板数据
    └── 生成dashboard_data.json
```
